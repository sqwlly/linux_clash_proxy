# 状态输出套件统一设计

**目标**

统一 `status`、`ai-status`、`list-groups`、`list-nodes`、`test-group`、`test` 的终端输出风格，让这些高频命令都具备一致的扫读体验。

**现状问题**

- Python CLI 与 `proxy.sh` 的标题、间距、区块风格不一致。
- 相同类型的信息，在不同命令里使用了不同表达方式，例如：
  - `=== 标题 ===`
  - 裸标题
  - 表头式输出
- 用户在切换命令时需要重新适应格式，认知成本偏高。

**设计原则**

- KISS：只统一结构和文案，不引入复杂 TUI 效果。
- DRY：复用相同的标题、状态词和结果行格式。
- YAGNI：不扩展命令能力，只改展示层。
- 结论前置：第一屏先给摘要，再给明细。

**统一风格**

所有默认输出统一为两层：

1. `摘要`
   - 给出最关键的结论，例如：
   - 当前状态
   - 当前出口
   - 可用数
   - 探测结果

2. `明细`
   - 给出支撑摘要的结构化信息，例如：
   - 链路
   - 候选列表
   - 检查结果
   - 配置路径
   - 资源信息

统一规则：

- 使用短标题，如 `摘要`、`链路`、`结果`、`资源`、`路径`
- 状态词统一为 `正常 / 异常 / 未运行 / 未知 / 部分异常`
- 成功失败行统一为 `正常  项目  细节` / `失败  项目  细节`
- `--raw` 保持脚本友好，不引入美化

**命令级设计**

`status`

- `摘要`：状态、API、AI 路由模式、AI 当前出口、运行配置状态
- `资源`：代理端口、控制接口、连接数、运行时间、内存、日志大小
- `路径`：原始配置、运行配置、实际运行配置、PID

`ai-status`

- `摘要`：AI 路由、AI 探测
- `连通性`：ChatGPT Web、OpenAI API
- `链路`：当前链路
- `备用`：备用路径
- `分组`：AI-MANUAL / AI-AUTO / AI-US / AI-SG

`list-groups`

- `摘要`：组数量、可切换组数量
- `列表`：组名、类型、当前选择

`list-nodes`

- `摘要`：目标组、当前选择、候选数
- `列表`：当前项与候选项

`test-group`

- `摘要`：目标组、可用、最佳、最慢
- `结果`：逐项结果

`test`

- `摘要`：目标、可用、出口 IP
- `结果`：GeoIP、URL 探测、出口 IP

**实现边界**

- 同时更新 Python CLI 与 `proxy.sh`，避免双入口风格漂移。
- 先补回归测试，再改实现。
- 文档同步更新 README / USAGE / TROUBLESHOOTING。

**非目标**

- 不引入 ANSI 复杂配色策略
- 不引入终端宽度自适应布局
- 不修改 `--raw` 语义
